**hooks/useRealtimeFeed.ts:**

```typescript
export function useRealtimeFeed(): {
  newPostCount: number;
  newPostIds: React.MutableRefObject<string[]>;
  showNewPosts: () => void;
}
```

Implementation:
1. Import `useEffect, useRef, useState, useCallback` from react, `useQueryClient, InfiniteData` from @tanstack/react-query, `getOrCreateChannel, removeChannel` from `@/lib/realtime`, `Content, ContentWithAuthor` from `@/types/database`.
2. Channel name: `'feed:content'`
3. In useEffect (deps: `[queryClient]`):
   - Call `getOrCreateChannel('feed:content')`
   - Chain `.on('postgres_changes', { event: 'UPDATE', schema: 'public', table: 'content', filter: 'status=eq.published' }, callback)`:
     - Extract `payload.new` as `Content`
     - Call `queryClient.setQueryData<InfiniteData<ContentWithAuthor[]>>(['feed'], ...)` to update `likes_count` and `comments_count` for the matching item. Use the exact same page-mapping pattern from feed.tsx optimistic updates. Only update `likes_count` and `comments_count` (not other fields) to avoid overwriting author data.
   - Chain `.on('postgres_changes', { event: 'INSERT', schema: 'public', table: 'content', filter: 'status=eq.published' }, callback)`:
     - Push `payload.new.id` into `newPostIds.current` ref array
     - Increment `newPostCount` state via `setNewPostCount(c => c + 1)`
     - Do NOT prepend to cache -- the banner (Plan 03) handles that
   - Chain `.on('postgres_changes', { event: 'DELETE', schema: 'public', table: 'content' }, callback)`:
     - Extract `payload.old.id` (only PK available on DELETE by default)
     - Filter the item out of all pages in the feed cache via `setQueryData`
   - Call `.subscribe((status, err) => { if (__DEV__) console.log('[Realtime] feed:content status:', status, err); })`
4. Cleanup: `return () => removeChannel('feed:content')`
5. `showNewPosts` callback: calls `queryClient.invalidateQueries({ queryKey: ['feed'] })` to refetch feed with new posts included, then resets `newPostCount` to 0 and clears `newPostIds.current`. We invalidate instead of manually prepending because INSERT payload lacks the joined author data needed for ContentWithAuthor.
6. Add 100ms debounce for UPDATE events to batch rapid-fire likes on viral posts. Use a `pendingUpdates` Map and `setTimeout` pattern as shown in RESEARCH.md "Debounce Pattern for Rapid-Fire Events". The debounce batches multiple updates within 100ms into a single `setQueryData` call.

**hooks/useRealtimeLikes.ts:**

```typescript
export function useRealtimeLikes(userId: string | undefined): void
```

Implementation:
1. Channel name: `'feed:likes'`
2. In useEffect (deps: `[userId, queryClient]`):
   - Early return if `!userId`
   - `getOrCreateChannel('feed:likes')`
   - `.on('postgres_changes', { event: 'INSERT', schema: 'public', table: 'likes', filter: \`user_id=eq.${userId}\` }, callback)`:
     - Extract `payload.new.content_id`
     - Update feed cache: set `is_liked: true` for matching item
   - `.on('postgres_changes', { event: 'DELETE', schema: 'public', table: 'likes', filter: \`user_id=eq.${userId}\` }, callback)`:
     - Extract `payload.old.content_id` (available because content_id is part of composite PK)
     - Update feed cache: set `is_liked: false` for matching item
   - `.subscribe()`
3. Cleanup: `removeChannel('feed:likes')`
4. Note: The filter `user_id=eq.${userId}` means we only get events for the CURRENT user's likes. This confirms/corrects the optimistic update already done by the like mutation in feed.tsx.

**hooks/useRealtimeComments.ts:**

```typescript
export function useRealtimeComments(): void
```

Implementation:
1. Channel name: `'feed:comments'`
2. In useEffect (deps: `[queryClient]`):
   - `getOrCreateChannel('feed:comments')`
   - `.on('postgres_changes', { event: 'INSERT', schema: 'public', table: 'comments' }, callback)`:
     - Extract `payload.new.content_id`
     - Update feed cache: increment `comments_count` by 1 for matching item
     - Also invalidate `['comments', payload.new.content_id]` query so the comments modal refreshes if open
   - `.subscribe()`
3. Cleanup: `removeChannel('feed:comments')`
4. Note: We subscribe to INSERT only (not UPDATE/DELETE) since comments_count on the content row is already tracked by the useRealtimeFeed UPDATE handler. This hook provides faster comments_count updates by reacting to the comment INSERT directly rather than waiting for the content row UPDATE trigger. If both fire, the content UPDATE (which is authoritative from the DB trigger) will correct any drift.
