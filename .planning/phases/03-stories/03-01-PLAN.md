Create migration that:

1. Creates a `cleanup_expired_stories()` SECURITY DEFINER function that:
   - Finds all stories where `expires_at <= NOW()`
   - For each expired story, deletes the storage object using `storage_path` (not URL parsing)
   - Deletes the expired story rows (CASCADE will handle story_views)

2. Schedules the function via pg_cron to run hourly at minute 0.

```sql
-- Migration 036: pg_cron story expiry cleanup

-- Enable pg_cron if not already (Supabase Pro has it available)
CREATE EXTENSION IF NOT EXISTS pg_cron;

-- Cleanup function
CREATE OR REPLACE FUNCTION cleanup_expired_stories()
RETURNS void
LANGUAGE plpgsql
SECURITY DEFINER
AS $$
DECLARE
  expired RECORD;
BEGIN
  FOR expired IN
    SELECT id, storage_path
    FROM stories
    WHERE expires_at <= NOW()
  LOOP
    -- Delete storage object if path is known
    IF expired.storage_path IS NOT NULL THEN
      DELETE FROM storage.objects
      WHERE bucket_id = 'stories'
        AND name = expired.storage_path;
    END IF;
  END LOOP;

  -- Delete expired story rows (CASCADE deletes story_views)
  DELETE FROM stories WHERE expires_at <= NOW();
END;
$$;

-- Schedule hourly at minute 0
SELECT cron.schedule(
  'cleanup-expired-stories',
  '0 * * * *',
  'SELECT cleanup_expired_stories();'
);
```

Important: Use `storage_path` column (from Task 1) instead of parsing `media_url`. If `storage_path` is NULL for legacy rows, skip the storage delete (the row delete still happens).
